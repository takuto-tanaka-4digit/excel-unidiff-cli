CLI インターフェイス設計（args/flags/subcommands/help/output/errors/config）の実用ルーブリック。人間優先を保ちつつ、合成可能性とスクリプト性を維持する。

## 序文

- CLI は依然として強力: システムの検査/制御ができ、対話・自動化の両方で動く。
- 現代 CLI = 人間優先のテキスト UI。機械向け REPL の薄い皮ではない。
- 目的: 有用性 + アクセシビリティ最大化; 人間と合成を設計対象にする。

## イントロダクション

- 本ガイドは思想 + 具体ルールを混在; 理論より例を重視。
- 対象外: フルスクリーン TUI（vim/emacs 的）。
- 言語/ツール非依存: 実装スタックに関わらず適用。

## 哲学

### 人間優先デザイン

- 既定は人間最適化; スクリプトは安定モード（`--json`, `--plain`, exit codes）で維持。
- 開発者向け出力を通常ユーザーに漏らさない; verbose/debug に限定。

### 組み合わせて動く単純な部品

- あなたの出力は誰かの入力になる前提。
- stdio/exit codes/signals を尊重; 主出力は stdout。
- パイプには行志向のプレーンテキストを優先; 構造化は JSON 追加。

### プログラム間の一貫性

- 使いやすさを損なわない限り共通慣習に従う。
- 標準フラグ名を再利用（`--help`, `--version`, `--json`, `--dry-run`, …）。

### （ちょうど）十分に伝える

- 少なすぎると「無言で固まる」。多すぎるとノイジーなデバッグ吐き出し。
- 進捗/状態は見せるが、成功時の出力は短く。

### 発見しやすさ

- ヘルプは UX の一部。例を先に; 次のコマンドを提案。
- ユーザーが間違えたら回復を助ける: 正しい構文/フラグへ導く。

### 会話を前提にする

- 試行錯誤のループを前提にする。繰り返し呼び出しに耐える設計。
- 安全な「dry run」/プレビュー、途中状態の提示、怖い操作の確認。

### 堅牢性

- 正しいだけでなく“堅牢に感じる”こと: 応答性、明快さ、デフォルトで怖いスタックトレースを出さない。
- 不正入力を丁寧に扱う; 早期検証; 明確なエラー。

### 共感

- ユーザーの味方になる。成功しやすく、失敗は情報豊富に。
- 個性は OK; ただし散らかすな。

### 混沌

- 端末の世界は不一致だらけ; 規範に従いつつ、必要なら意図的に破る。
- 逸脱するなら明確にし、文書化する。

## ガイドライン

### 基本

- 可能なら実績ある引数パーサー（組み込み/OSS）を使う。
- 退出コード: 成功は `0`、失敗は非 0; 重要な失敗モードを数個にマップ。
- stdout は主出力（機械可読含む）。stderr はメッセージ/ログ/エラー。

### ヘルプ

- `-h`/`--help` を常にサポート。`-h` の別用途は不可。
- 必須引数が足りない場合は、簡潔なヘルプ + 1–2 例 + 「--help を使う」。
- Git 風 CLI: `mycmd help`, `mycmd help subcmd`, `mycmd subcmd --help` をサポート。
- サポート導線（repo/issues/docs）を示す。Web docs があるならサブコマンド単位の深いリンクを推奨。
- 例を先に; よく使うフラグ/コマンドを先に; エスケープだらけの読みづらさを避ける。

### ドキュメント

- Web docs を提供（検索可能/リンク可能）。
- 端末内 docs（`mycmd help ...`）を提供; 必要なら man ページ。

### 出力

- 人間優先、機械は次: TTY を検出して書式を切替。
- 人間向け出力が解析を壊すなら `--plain`（安定な行ベース）や `--json` を用意。
- 成功時は通常 “何か” を出すが簡潔に; 必要なら `-q/--quiet`。
- 状態を変えたら、何が変わったかと新状態を伝える。
- ワークフロー型ツールでは “次のコマンド” を提案。
- 色は控えめに; TTY でない/`NO_COLOR`/`TERM=dumb`/`--no-color` では無効。
- stdout が TTY でない場合、アニメーション/プログレスバーは禁止。
- 長い出力は対話時のみ pager を使う; `less` の一般設定: `-FIRX`。

### エラー

- 予期されるエラーは人間向けに書き換える; デフォルトでスタックトレースは出さない。
- ノイズを抑える; 繰り返しエラーはまとめる。
- 重要情報は最後に; 赤文字は意図的に（ユーザーを赤で溺れさせない）。
- 予期しないクラッシュ: デバッグ情報 + バグ報告手順を示す; 大きなログはファイルへ。

### 引数とフラグ

- 明確さと将来性のため、位置引数よりフラグを優先。
- すべてのフラグに長い形式を用意; 1 文字フラグは最頻用のみ。
- 同種の繰り返し項目は複数引数 OK（`rm a b c`）; “異種の位置引数が2つ以上” は避ける。
- 標準フラグ名（共通セット）:
  - `-h, --help` help
  - `--version` version
  - `-q, --quiet` less output
  - `-v, --verbose` more output（`-v` を version にしない）
  - `-d, --debug` debug output
  - `-f, --force` skip confirmation / force
  - `-n, --dry-run` preview only
  - `--json` structured output
  - `-o, --output <file>` output path
  - `--no-input` disable prompts
- 既定値は多くのユーザーに正しいものにする（全員がフラグを alias している想定はしない）。
- 入出力がファイルなら stdin/stdout を示す `-` をサポート。
- シークレットはフラグに載せない; `--password-file` や stdin を使う。
- パーサーが許すならフラグ/サブコマンドは順序非依存を優先。

### 対話性

- stdin が TTY の時のみプロンプト。
- `--no-input`: 絶対にプロンプトしない; 必要入力が無ければ行動可能なエラーを返す。
- パスワードプロンプト: エコー無効。
- 退避路を明示（Ctrl-C や “press q” など）。

### サブコマンド

- 複雑さはサブコマンドで分離; グローバルフラグ/設定/ヘルプを共有。
- サブコマンド間で一貫性（命名、フラグ、出力、整形）。
- 名詞-動詞（`docker container create`）または動詞-名詞で統一。
- `update` と `upgrade` など曖昧ペアは避ける（明確に差別できる場合のみ）。
- “何でも受ける” サブコマンドは避ける; 任意の省略形は許さない（将来互換の罠）。

### 堅牢性

- 早期検証; 明確なエラーで即失敗。
- 応答性: <100ms で何か出す（特にネットワーク I/O 前）。
- 長時間タスクは進捗を出す（対話時のみ）; ログが交差して混乱しないようにする。
- ネットワーク呼び出しにタイムアウト; 設定可能にする。
- 再実行に強くする: 可能なら冪等、復旧可能、可能なら crash-only。

### 将来互換

- インターフェイスは契約: args/flags/subcommands/config/env vars/output modes。
- 変更は追加的に; 非推奨は早めに強く知らせ、移行手段を提供。
- 人間向け出力は進化させ、スクリプトは `--plain`/`--json` で安定させる。

### シグナルと制御文字

- Ctrl-C: 速やかに終了し、即座に何か伝える; 後始末は最小限。
- 2 回目の Ctrl-C: 強制終了を許可してもよい; 何をするかを伝える。
- 後始末が走らない前提で、crash-only 復旧を設計。

### 設定

- 適切な仕組みを選ぶ:
  - 呼び出しごと: flags（場合によって env）。
  - ユーザー/マシン単位: flags + env; 必要なら config。
  - プロジェクト単位（チェックインするもの）: リポ内 config ファイル。
- 適用できるなら XDG base directories を使う。
- 優先順位（高 → 低）: flags > process env > project config > user config > system config。
- 他プログラムの設定を黙って変更しない; 同意を取る; 既存編集より新規ファイルを優先。

### 環境変数

- 命名: 大文字 + 数字 + アンダースコア; 値は単一行推奨。
- 関連する共通変数を尊重: `NO_COLOR`, `DEBUG`, `EDITOR`, `PAGER`, proxy 変数, `TERM`, `TMPDIR`, `HOME`, `COLUMNS/LINES`。
- `.env` はプロジェクト単位の非秘匿設定には有用; ただし完全な config システムにはしない。
- 既定でシークレットを env で受けない; ファイル/パイプ/ソケット/シークレット管理を優先。

### 命名

- コマンド名: シンプルで覚えやすい小文字; 過度に汎用な衝突は避ける。
- 短くても謎にしない; 打ちやすさは重要。

### 配布

- 可能なら単一バイナリ; そうでなければアンインストール可能なネイティブパッケージ。
- アンインストールを容易に; 手順を記載。

### 分析

- 明示的同意なしの送信は禁止; 何を/なぜ/どう/保持期間を説明。
- opt-in を優先; opt-out なら明確で無効化しやすく。
- 代替案: docs での計測、ダウンロード統計、ユーザーへのヒアリング。

### さらに読む

- POSIX Utility Conventions
- GNU Coding Standards（特に flags/help の慣習）
- 12 Factor CLI Apps
- Heroku CLI Style Guide
